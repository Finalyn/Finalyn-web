Lighthouse 95+ : le guide complet pour un site rapide en 2026

10 juin 2026 · 9 min de lecture

Si votre site met plus de 2,5 secondes à s'afficher sur mobile, vous perdez la moitié de vos visiteurs avant même qu'ils voient votre logo. Google le sait, vous référence en fonction, et vos concurrents — ceux qui investissent dans la performance — vous prennent du trafic chaque semaine.

Cet article est la méthode exacte qu'on applique chez Finalyn.web sur chaque site qu'on livre. Tous nos projets passent 95+ sur Lighthouse mobile, pas par chance, mais parce qu'on attaque chaque point dans le bon ordre. Voici lequel.

Pourquoi viser 95+ et pas 80 ?

Lighthouse note de 0 à 100 sur quatre axes : Performance, Accessibilité, Bonnes pratiques, SEO. La note Performance est la plus difficile parce qu'elle dépend des Core Web Vitals — trois métriques que Google utilise dans son ranking depuis 2021.

Concrètement :

  • 0 à 49 : votre site est lent. Google vous pénalise en référencement, votre taux de rebond est élevé, vos conversions chutent.
  • 50 à 89 : vous êtes dans la moyenne. Pas pénalisé, pas favorisé. Vos concurrents qui poussent à 90+ vous passent devant.
  • 90 à 100 : Google considère votre site comme "rapide". Vous gagnez en ranking, votre taux de conversion grimpe, votre SEO suit.

Pourquoi 95 et pas 90 ? Parce que la note Lighthouse varie de ±5 points selon la connexion lors du test. Viser 95 garantit qu'on reste systématiquement au-dessus de 90 dans la vraie vie.

Les Core Web Vitals : ce que Google mesure vraiment

LCP (Largest Contentful Paint) — viser < 2,5s

C'est le temps que met votre élément principal (souvent le titre H1 ou l'image hero) à apparaître. Si votre LCP dépasse 4 secondes, vous êtes en rouge dans le rapport et Google vous le fait sentir.

Les causes typiques d'un mauvais LCP :

  • Une image hero trop lourde (PNG de 3 Mo au lieu d'un WebP de 200 Ko)
  • Des fonts Google chargées sans preconnect
  • Du CSS bloquant en haut du <head>
  • Un serveur qui répond lentement (TTFB > 800 ms)

CLS (Cumulative Layout Shift) — viser < 0,1

Le CLS mesure combien votre page "saute" pendant le chargement. Vous avez sûrement vécu ça : vous lisez un article, une pub se charge, tout le contenu descend de 200 pixels, vous cliquez sur le mauvais bouton. C'est ça, le CLS.

Les coupables : images sans dimensions définies, polices web qui changent de taille pendant le chargement, contenus injectés dynamiquement (cookies, chat widgets, pubs).

INP (Interaction to Next Paint) — viser < 200 ms

L'INP a remplacé le FID en mars 2024. C'est le temps de réponse à une interaction (clic, tap, frappe clavier). Si votre menu burger met 500 ms à s'ouvrir, votre INP est mauvais.

L'INP punit le JavaScript bloquant : trop de scripts third-party (Google Analytics, Hotjar, Intercom, Calendly) chargés au mauvais moment, ou des animations CSS lourdes sur des appareils bas de gamme.

La méthode Finalyn pour passer 95+ en 5 étapes

1. Images : WebP + dimensions explicites + lazy loading

L'optimisation des images représente 60% du gain de performance sur un site moyen. Voici la règle d'or :

  • Convertissez tout en WebP (ou AVIF si supporté). Une image WebP pèse 25-35% de moins qu'un PNG équivalent.
  • Ajoutez systématiquement width et height sur chaque <img>. Sans ça, le navigateur ne réserve pas l'espace et votre CLS explose.
  • Utilisez loading="lazy" sur toutes les images sauf celle visible immédiatement (l'image hero). Cette dernière mérite parfois un fetchpriority="high".
  • Servez des tailles adaptées avec srcset. Un téléphone n'a pas besoin d'une image 4K.

2. Fonts : preload + font-display: swap

Les polices Google Fonts sont la deuxième cause de LCP médiocre. Sans optimisation, voici ce qui se passe : votre HTML charge → le CSS demande la font → le navigateur va chercher sur Google → la font arrive → tout se réorganise (CLS qui monte).

La solution :

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="...font CSS..." rel="stylesheet">

Et dans votre CSS, ajoutez font-display: swap pour que le texte s'affiche immédiatement avec une font système, puis bascule en douceur sur votre font web une fois chargée.

3. JavaScript : minimum vital, defer, et zéro framework si possible

Un site React/Next.js de base envoie au navigateur 150-300 Ko de JavaScript juste pour afficher du texte. Sur 4G suisse, c'est encore tolérable, mais pour un site vitrine ou une landing, c'est du sur-engineering qui plombe votre Lighthouse.

Notre règle : HTML/CSS/JS vanilla pour tous les sites où on n'a pas besoin de state complexe. Quand React est justifié (app SaaS, dashboard métier), on optimise avec code-splitting, lazy routes et server components.

Dans tous les cas, mettez defer ou async sur tous vos <script> non-critiques.

4. CSS : inline ce qui est above-the-fold

Le CSS bloque le rendu. Tant que votre fichier styles.css n'est pas chargé, le navigateur n'affiche rien. Pour un LCP rapide, deux approches :

  1. Critical CSS inliné : on extrait les ~10 Ko de CSS nécessaires au premier écran et on les met directement dans <style> en haut du <head>. Le reste charge en différé.
  2. CSS unique et léger : un seul fichier < 50 Ko bien organisé. Pas de Tailwind compilé non purgé, pas de Bootstrap entier. Ce qu'on n'utilise pas, on ne le charge pas.

5. Hébergement : un CDN, pas un mutualisé OVH

Votre site peut être parfaitement optimisé, si votre serveur met 1,5 seconde à répondre au premier byte (TTFB), votre LCP sera mauvais. Investissez dans un hébergement moderne.

Nos recommandations 2026 pour les PME suisses :

  • Sites vitrines / e-commerce light : Vercel ou Netlify (gratuit jusqu'à un certain trafic, CDN mondial inclus)
  • E-commerce conséquent : Infomaniak Jelastic (données en Suisse, RGPD/nLPD nativement)
  • SaaS / apps lourdes : Cloudflare Pages + Workers pour le edge, Supabase pour la base données

Évitez les mutualisés bas de gamme (Hostinger à 2 €/mois, OVH Web Perso). L'économie de 5 €/mois vous coûte 30% de conversion.

"Un site rapide n'est pas un luxe technique, c'est un investissement direct dans votre chiffre d'affaires. Amazon avait calculé qu'une seconde de latence en plus coûtait 1,6 milliard de dollars par an. Votre PME est plus petite, mais le ratio est le même."

Le piège des outils third-party

Chaque widget ajouté à votre site est un risque de performance. Voici les vrais coûts :

OutilPoids JSImpact LCP
Google Analytics~45 KoFaible si bien chargé (defer + idle)
Calendly inline~120 KoLourd, à éviter en visible
Intercom chat~250 KoTrès lourd, charger après interaction
Hotjar / Clarity~80 KoMoyen, charger en différé
Stripe Checkout~90 KoCharger uniquement sur les pages panier

Notre règle : on demande systématiquement "est-ce vraiment nécessaire ?" avant d'intégrer un widget. Et quand on l'intègre, on le charge avec requestIdleCallback ou après une interaction utilisateur.

Comment tester (et continuer à tester après livraison)

Trois outils complémentaires :

  • PageSpeed Insights : le test de référence, données mobiles réelles (CrUX)
  • Chrome DevTools → onglet Lighthouse : pour des tests rapides en local
  • WebPageTest : tests depuis plusieurs zones géographiques, idéal pour vérifier le rendu depuis Zürich, Genève, Lugano

Mettez en place une vérification mensuelle. Un site qui passe 95 à la livraison peut tomber à 75 six mois plus tard si on a ajouté trois widgets sans réfléchir.
